# Intelligent Cutovers — Runtime Scope

Companion to the platform scope at
[`docs/active/intelligent-cutovers-scope.md`](https://github.com/kmgrassi/parallel-agent-platform/blob/main/docs/active/intelligent-cutovers-scope.md)
in `parallel-agent-platform`. That doc owns the design (model-tier
registry, execution-profile shape, routing-rule columns, audit table,
behavior contract). This doc owns the **runtime implementation**: the
cutover engine module, the integration points into each runner, the
cooldown tracker, and the audit-row writes.

Read the platform scope first. This doc assumes its vocabulary
(`fallbacks`, `modelTierFloor`, `provider_cutover` table, the cutover
trigger code set).

## Goal

Inside the orchestrator, when a provider call fails with a
cutover-eligible error code, the runtime should walk the execution
profile's fallback chain — skipping links below the adequacy floor —
until a link succeeds or the chain is exhausted. Every decision writes
an audit row.

Specifically:

1. **A single `Cutover` module** that owns the walk. Runners do not
   each implement their own fallback logic; they delegate to the
   engine.
2. **Integration at every provider call site** — Codex, OpenClaw,
   ComputerUse, LLM tool runner, LocalRelay. Each catches its own
   errors (already does), classifies them via `Observability` (already
   does), and hands a cutover-eligible failure to the engine instead of
   propagating it.
3. **A cooldown tracker** keyed by `(workspace_id, credential_id)` to
   avoid re-trying a known-rate-limited credential for a short window.
   In-memory only; not persisted.
4. **Audit-row writes** through a thin repository to `provider_cutover`
   via the platform's REST surface (PostgREST bridge), matching the
   pattern other runtime → DB writes use.
5. **No double-billing on observability.** A cutover-triggered failure
   should still write one `RuntimeLog` event per attempt (we keep the
   existing observability shape) AND one `provider_cutover` row per
   *decision* (cutover or escalation). The cutover row is the decision
   record; the runtime logs are the attempt records.

## Current state

### Where provider calls happen

- `apps/orchestrator/lib/symphony_elixir/runner/openclaw.ex:84-100` —
  HTTP `POST /v1/runs`, classifies HTTP errors via
  `Observability.provider_status_failure/2`, returns
  `{:error, {:retryable, ...}}` on failure.
- `apps/orchestrator/lib/symphony_elixir/runner/llm_tool_runner.ex:126`
  — `model_client_create_response(session, request, attempt)` calls
  through `session.model_client.create_response()` behaviour. Returns
  raw error to `run_model_loop` (`:126`), which fails the turn.
- `apps/orchestrator/lib/symphony_elixir/runner/codex.ex` — subprocess
  via `AppServer.*`, JSON-RPC. Provider errors surface as RPC error
  payloads; classification is opaque at this level.
- `apps/orchestrator/lib/symphony_elixir/runner/computer_use.ex` — same
  shape as OpenClaw (HTTP polling).
- `SymphonyElixir.Runner.LocalRelay` — relay socket; errors come back
  framed in the relay protocol.

### What `Observability` already does

`apps/orchestrator/lib/symphony_elixir/runner/observability.ex`:

- `:15-30` — `@retryable_provider_codes`: the existing list of codes
  that are *eligible* for retry.
- `:108-136` — `provider_status_failure/2` returns a struct with
  `{error_code, retryable, status_code, body_excerpt, ...}`.
- `:167-171` — `log_provider_failure/2` writes one structured
  `RuntimeLog` event per attempt.
- `:225-245` — HTTP status → error code mapping.

The cutover engine consumes the output of `provider_status_failure/2`
directly. No new classification needed; only the new
`provider_content_refused` code (added by Phase 2 of the platform
scope).

### What does NOT exist in the runtime

- No central retry / fallback module.
- No cooldown tracker.
- No persistence of cutover decisions.
- The `attempt` counter in `llm_tool_runner.ex:126` exists but is not
  incremented anywhere — there is no second-attempt code path.
- No reader for `ExecutionProfile.fallbacks` (the field doesn't exist
  yet; Phase 3 of platform scope adds it).
- `apps/orchestrator/lib/symphony_elixir/execution_profile.ex` is the
  Elixir mirror of the contract — it gets regenerated by
  `pnpm run supabase:schema:sync` and currently has no fallback fields.

## Proposed model

### New module: `SymphonyElixir.Cutover`

`apps/orchestrator/lib/symphony_elixir/cutover.ex` — single GenServer
process per orchestrator instance, owning the cooldown table. Public
API:

```elixir
@doc """
Walk the execution profile's fallback chain for a provider call.

`call_fn` is a 1-arity function: it receives a `%CutoverLink{provider,
model, credential_ref}` and must return `{:ok, result}` or
`{:error, %Observability.ProviderFailure{}}`.

Returns `{:ok, result, %CutoverDecision{}}` or
`{:error, :exhausted | :floor_exhausted, %CutoverDecision{}}`.
"""
@spec walk(ExecutionProfile.t(), context :: map(), (CutoverLink.t() -> term())) :: term()
def walk(profile, context, call_fn)
```

Behavior — directly mirrors the platform doc's contract:

1. Compose the chain: `[primary | fallbacks]`.
2. For each link in order:
   - Look up `MODEL_TIER_REGISTRY` via `ModelTiers.tier_of/2`. If below
     `profile.model_tier_floor`, skip; record skip reason.
   - Check cooldown: if `(workspace_id, link.credential_id)` is in the
     cooldown set, skip; record skip reason.
   - Invoke `call_fn.(link)`. On `{:ok, result}` — return success with
     a `CutoverDecision` row built from the walk so far.
   - On `{:error, %ProviderFailure{retryable: true}}` — record the
     attempt, mark credential in cooldown (if 429), continue.
   - On `{:error, %ProviderFailure{retryable: false}}` — stop walking,
     return the error; the caller decides whether to fail the task or
     escalate to a human via Pillar 4.5.
3. If the chain exhausts:
   - If any link was skipped due to floor, return `:floor_exhausted`.
   - Otherwise return `:exhausted`.
   - Caller pushes to attention queue with appropriate reason.

### Integration: how each runner uses the engine

Every provider call site changes from:

```elixir
# BEFORE
case post(session, "/v1/runs", body) do
  {:ok, resp} -> ...
  {:error, failure} -> {:error, {:retryable, failure}}
end
```

to:

```elixir
# AFTER
Cutover.walk(session.execution_profile, %{work_item_id: session.work_item_id}, fn link ->
  session_for_link = put_link(session, link)
  case post(session_for_link, "/v1/runs", body) do
    {:ok, resp} -> {:ok, resp}
    {:error, _http_error} = err ->
      failure = Observability.provider_status_failure(err, link)
      {:error, failure}
  end
end)
|> case do
  {:ok, result, _decision} -> result
  {:error, :exhausted, decision} ->
    Attention.escalate(:cutover_exhausted, decision, session)
  {:error, :floor_exhausted, decision} ->
    Attention.escalate(:cutover_floor_exhausted, decision, session)
end
```

The runner does not itself walk the chain; it provides the per-link
call closure and lets `Cutover.walk/3` drive.

### Cooldown tracker

In-memory ETS table owned by `SymphonyElixir.Cutover` GenServer:

```elixir
:ets.new(:cutover_cooldowns, [:set, :protected, :named_table])
# rows: {{workspace_id, credential_id}, expires_at}
```

- On a 429, insert with `expires_at = now + 60s` (configurable via
  `config/runtime.exs`).
- Other retryable codes do not set a cooldown (only persistent
  rate-limit signal does — a 5xx is transient and we retry promptly).
- Cooldown lookup is O(1); pruning is lazy (check `expires_at` on
  read, delete if past).
- Forgotten on orchestrator restart (acceptable — per OQ-CU-2 in
  platform scope).

### Audit-row writes

A new repository module
`apps/orchestrator/lib/symphony_elixir/cutover/audit.ex`:

```elixir
@spec write(CutoverDecision.t()) :: :ok | {:error, term()}
def write(decision)
```

Posts to PostgREST `/rest/v1/provider_cutover` with the row payload.
Uses the existing best-effort persistence pattern (see
[`best-effort-persistence-logging.md`](./best-effort-persistence-logging.md))
— audit failures are logged but do not fail the cutover walk. The
walk's success is the source of truth; the audit row is observability.

Schema sync (`pnpm run supabase:schema:sync`) must include
`provider_cutover` in the `BRIDGE_TABLES` list in
`scripts/append-supabase-jsdoc-types.mjs` (called out in the
runtime's CLAUDE.md as a known gotcha).

### Helper-side participation

When the helper relays a local-model failure, the failure frame must
carry a `provider_*` error code matching the runtime's vocabulary so
that `Cutover.walk/3` treats it the same as a cloud failure.

- `apps/orchestrator/lib/symphony_elixir/runner/local_relay.ex` —
  parses the relay frame; today the error payload is opaque.
- Extend relay protocol (see
  [`local-relay-protocol.md`](./local-relay-protocol.md)) to include
  `error_code: "provider_overloaded" | "provider_timeout" | ...` on
  failure frames.
- Helper side (Go): `internal/runner/openai_compatible/` already
  catches local Ollama errors; it must emit them as the canonical
  codes rather than bare strings.

Helper changes are small — a section of the
[helper PR plan](https://github.com/kmgrassi/local-runtime-helper/blob/main/docs/local-runtime-helper-pr-plan.md)
rather than a dedicated helper-side scope.

### `ExecutionProfile` Elixir mirror

`apps/orchestrator/lib/symphony_elixir/execution_profile.ex` gets two
new fields when the contracts file is regenerated:

```elixir
defstruct [
  :agent_id, :workspace_id, :role, :runner_kind,
  :provider, :model, :credential_ref,    # primary
  fallbacks: [],                         # new
  model_tier_floor: "any",               # new
  :tool_profile, ...
]
```

`fallbacks` is a list of `%CutoverLink{}` structs. The Elixir mirror
gets the same validation as the TS contract (e.g. each fallback's
`model` must exist in `MODEL_TIER_REGISTRY`).

Runtime-side phases interleave with platform phases (numbered in
platform scope). This is a runtime-only view.

  `apps/orchestrator/lib/symphony_elixir/model_tiers.ex`, generated
  from `contracts/model-tiers.ts` by the schema-sync script.
- `ModelTiers.tier_of(provider, model) :: :frontier | :mid | :local | nil`.
- `ModelTiers.meets_floor?(model_tier, floor)` helper.
- No behavior change; just the lookup.

### R-2 — Detect `provider_content_refused`

Tracks platform Phase 2.

- Extend `observability.ex` `@retryable_provider_codes`.
- Per-provider detection helpers in each runner:
  - OpenClaw / ComputerUse: inspect HTTP body for content-policy 4xx.
  - LLM tool runner: inspect response for Anthropic `refusal` blocks /
    OpenAI `finish_reason="content_filter"`.
  - Codex: inspect RPC error payloads for refusal codes.
- New unit tests in `test/symphony_elixir/runner/observability_test.exs`.

### R-3 — Add `Cutover` module + cooldown ETS table

- New `apps/orchestrator/lib/symphony_elixir/cutover.ex` GenServer.
- Supervisor wiring in `apps/orchestrator/lib/symphony_elixir/application.ex`.
- Tests: unit tests with a stub `call_fn` for floor logic, cooldown
  behavior, exhaustion behavior.

### R-4 — Wire `Cutover.walk/3` into LLM tool runner

LLM tool runner is the highest-traffic call site and the most
straightforward HTTP shape.

- Refactor `llm_tool_runner.ex:126` `model_client_create_response` to
  delegate to `Cutover.walk/3`.
- Wire `Attention.escalate/3` (placeholder if Pillar 4.5 attention
  queue isn't built yet — see open questions).

### R-5 — Wire into OpenClaw and ComputerUse

Same shape as LLM tool runner. Likely the smallest diff per runner.

### R-6 — Wire into Codex

Trickier because Codex errors come via JSON-RPC; need to map RPC
error payloads to `Observability.ProviderFailure` shapes. May require
a small `Codex.classify_error/1` helper.

### R-7 — Wire into LocalRelay + extend relay protocol

Helper side (Go) emits canonical error codes; runtime LocalRelay
parses them; cutover treats local failures identically to cloud.

### R-8 — Cutover audit row writes

Tracks platform Phase 5.

- New `apps/orchestrator/lib/symphony_elixir/cutover/audit.ex`.
- Add `provider_cutover` to `BRIDGE_TABLES` in
  `scripts/append-supabase-jsdoc-types.mjs`.
- `Cutover.walk/3` emits an audit row at end of walk (success or
  failure).

## Open questions

### OQ-CR-1 — `Attention.escalate/3` target — RESOLVED

**Resolved** by
[`policy-trust-dial-runtime-scope`](./policy-trust-dial-runtime-scope.md).
That scope's Phase R-2 builds the real `Attention` module that writes
to the `escalation` table (created by Pillar 4.6's platform Phase 2).

Cutover engine integration: `Attention.escalate(:cutover_exhausted,
decision, session)` and `Attention.escalate(:cutover_floor_exhausted,
decision, session)` write `trigger_kind: "resource"` rows with
`trigger_detail` carrying the cutover decision. No call-site changes
on the cutover side — the placeholder note disappears as soon as the
policy-trust-dial runtime work lands.

### OQ-CR-2 — Per-orchestrator vs per-workspace cooldown

The cooldown table is keyed by `(workspace_id, credential_id)` but
lives in a per-orchestrator-instance ETS table. If a workspace runs
multiple agents across multiple orchestrator instances, each instance
has its own cooldown view.

**Tentative answer**: per-instance is fine. Stale cooldown across
instances is conservative (one instance may try a credential the other
just learned was rate-limited; the second attempt will 429 again and
re-set the cooldown). Persisting to Redis is over-engineering for the
expected volume.

### OQ-CR-3 — Cutover during a tool-call sequence mid-turn

A turn may contain multiple model calls (the agent calls a tool, the
tool returns, the model is invoked again to react). If the first model
call succeeds but the second 429s, do we cut over on just the second
call, or restart the whole turn?

**Tentative answer**: cut over on the second call. The conversation
state up to that point is sound; only the in-flight assistant turn is
discarded. Carry-over semantics (per platform scope) cover this case.

### OQ-CR-4 — Attempts counter

`llm_tool_runner.ex:126` has an unused `attempt` parameter. Does the
cutover engine increment it? Does the runtime log carry attempt
numbers across links of the chain (`attempt=1` on the primary,
`attempt=2` on first fallback)?

**Tentative answer**: yes — attempt counter advances across the walk
so the runtime log timeline can be reconstructed. Each `RuntimeLog`
event for a cutover walk records `attempt` and `cutover_id` so the
audit row and the per-attempt logs can be joined.

## Out of scope

- **Same-process retry without cutover** (e.g. retrying a single 503
  three times before giving up on a link). The walk goes straight to
  the next link on any cutover-eligible failure. If you want multiple
  attempts at the same link, list the link twice in the chain — but
  that's almost always wrong because the same key just hit 429.
- **Cross-orchestrator cooldown sharing** (see OQ-CR-2).
- **A separate retry budget per task.** The chain length is the budget;
  no second "max retries" knob.
- **Partial-output stitching across cutover.** Discard partial output,
  replay user message (per platform OQ-CU-4).
- **Surfacing cutover badges in the dashboard.** Platform Phase 5 owns
  the API; web UI changes are in platform Phase 6.

## Success criteria

1. `Cutover.walk/3` exists and is the only place in the runtime that
   walks a fallback chain. No runner has its own retry logic.
2. Every provider call site (Codex, OpenClaw, ComputerUse, LLM tool
   runner, LocalRelay) routes its failures through the engine.
3. A 429 against the primary triggers a single audit row, a successful
   call against fallback link 1, and one `RuntimeLog` event per
   attempt with matching `cutover_id`.
4. An execution profile with `model_tier_floor: "frontier"` and a
   chain that includes a `mid` model never invokes the mid model. The
   audit row records `escalated_floor` if no other link satisfies the
   floor.
5. The cooldown table holds entries for known-rate-limited credentials
   and the engine skips them for the configured window. No persistence
   across restart; behavior on restart is "retry, learn again."
6. Helper-side local-model failures arrive at the runtime with
   canonical `provider_*` error codes and are walked through the
   chain identically to cloud failures.

When these are true alongside the platform success criteria, Pillar
3.4 closes.
