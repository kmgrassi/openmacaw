# Learning Sidecar Scope

A Hermes-style learning layer adapted for the parallel-agent stack
(parallel-agent-platform + parallel-agent-runtime + harper-server).
Workspace-scoped rather than personal, advisory rather than self-modifying,
and built on top of infrastructure that's already provisioned.

Companion: [`learning-sidecar-pr-plan.md`](./learning-sidecar-pr-plan.md)
sequences the implementation into PRs across all three repos.

Cross-reference: this scope assumes
[`agent-skills-platform-scope.md`](./agent-skills-platform-scope.md) is
either landed or landing in parallel. The skill-distillation track of
this scope depends on the skills table/contract from that scope; the
memory tracks do not.

## Why

Hermes (Nous Research) is a self-improving personal-agent runtime: it
curates memories, summarizes prior sessions, distills repeated successful
actions into reusable skills, and feeds that context back into future runs.

The parallel-agent stack today does the orchestration half (Linear/work-item
intake → workspace → agent run → terminal outcome) but does not retain or
reuse anything across runs. Each run starts cold. Repos, tasks, and review
expectations that recur across dozens of runs have no memory.

The Hermes blueprint translates almost directly with one substitution:

> Hermes learns about a user. Symphony/parallel-agent should learn about
> the workspace — the repo, the workflow, CI quirks, code-review
> expectations, flaky tests, deployment processes, and recurring task
> patterns. Memory keys are workspace + agent + agent_type, not user.

Symphony's README explicitly frames the current implementation as preview
software for trusted environments. We mirror that framing: the learning
sidecar must remain auditable and never silently mutate workflow
instructions. Skill candidates surface as PRs for human review, not as
auto-merged file writes.

## Existing foundation (what we don't need to build)

The audit done while scoping this work turned up surprisingly complete
prior art. Each of these exists today and is in the migration history,
even though almost none of it is wired into production code paths yet.

### Storage

`harper-server` migration `20260227124000_create_memory_items_and_search.sql`
provisioned the `memory_items` table with exactly the shape we need:

```
public.memory_items (
  id              uuid PK
  workspace_id    uuid NOT NULL  -- workspace scope (RLS-enforced)
  agent_id        uuid NULL      -- optional per-agent narrowing
  scope           memory_scope   -- enum: long_term | daily | project
                                 --       | run_summary | scratch
  content         text NOT NULL
  tags            jsonb
  importance      int
  event_time      timestamptz
  source_run_id   text NULL      -- link back to run that produced it
  source_task_id  text NULL      -- link back to task / work item
  canonical_id    uuid NULL      -- soft-merge target
  supersedes_id   uuid NULL      -- prior version this replaces
  is_deleted      boolean
  embedding       vector(1536)   -- pgvector, OpenAI dim
  fts             tsvector       -- generated by trigger
  created_at, updated_at
)
```

Indexes already cover `(workspace_id)`, `(workspace_id, agent_id)`,
`(workspace_id, scope)`, `(workspace_id, event_time DESC)`. RLS policies
landed in `20260227125000_apply_workspace_memory_rls.sql`.

### Retrieval

The same migration ships `public.memory_hybrid_search(...)` — a single SQL
function that takes a workspace ID, optional agent ID + scope filter, a
text query, and an embedding, then returns RRF-ranked rows combining
**FTS** (Postgres full-text search: `tsvector` / `tsquery` via the
trigger-maintained `memory_items.fts` column) and **pgvector cosine
similarity** on the `embedding` column. Hermes-style "search across
prior sessions" works out of the box once anything starts writing rows.
When the workspace has no embedding provider configured, the function
gracefully degrades to FTS-only ranking.

### Run lifecycle hook

`parallel-agent-runtime/apps/orchestrator/lib/symphony_elixir/broker_log_adapter.ex`
exposes `BrokerLogAdapter.finalize/2`, called by `AgentRunner` when a run
ends. At that hook we have: `agent_id`, `workspace_id`, `run_id`,
`session_thread_id`, the full `messages` transcript via `SessionStore`,
`status` (`completed | failed | cancelled`), `terminal_reason`, token
`usage`, final `output`, and `error` payload. That's the entire reflection
input set, ready without any new event plumbing.

### Scheduled-task framework

The orchestrator already polls a `scheduled_task` table on a configurable
tick and delivers prompts to agents via `ChatGateway`. A nightly
consolidation or skill-distillation job is just another row in that table
— no new scheduler needed.

### Skills (in-flight scope)

`docs/active/agent-skills-platform-scope.md` defines the skills table,
CRUD, and `/api/agents/:id/skills/resolve` endpoint. The skill-distillation
track of this scope produces *candidates* that flow into that system. We
do **not** redefine the skills surface here; we consume it.

### Gaps (what we actually need to build)

| Layer | Gap |
|---|---|
| Memory write | No production code inserts into `memory_items` |
| Memory read | No code reads `memory_hybrid_search` |
| Reflection | No code summarizes a finished run into memories |
| Retrieval injection | The runtime prompt builder doesn't fetch memories |
| Distillation | No process turns clusters of memories into skill candidates |
| Governance | No skill-candidate-via-PR pipeline |

## Design

```
       Linear / work_item / scheduled_task
                   │
                   ▼
        Platform API (/apps/api)
                   │  enrich prompt with retrieved memories
                   ▼
     Orchestrator (Elixir; AgentRunner)
                   │
              run executes
                   │
                   ▼
       BrokerLogAdapter.finalize/2  ◄──── existing hook
                   │
        ┌──────────┼──────────────┐
        ▼          ▼              ▼
   broker_run    SessionStore   Learning Sidecar  ◄── new
    (persist)    (in-mem)            │
                                     │
                            ┌────────┼──────────────────┐
                            ▼        ▼                  ▼
                       Reflector   MemoryWriter   (eventually)
                       (LLM)       (memory_items)  Skill Distiller
                                                       │
                                                       ▼
                                                 PR-bot opens
                                                 .codex/skills/<name>.md
```

Five logical components, each thin:

### 1. Memory Indexer

After every run terminal state, write 1–N `memory_items` rows with
`scope='run_summary'`, `source_run_id=<run>`, `source_task_id=<work_item>`,
`agent_id=<agent>`, `workspace_id=<workspace>`. Initially: one
summary row per run plus optional per-failure-mode rows. Embedding is
computed at write time (same provider as the agent's model, falls back
to the workspace's configured embedding credential).

Lives in: **parallel-agent-runtime** (Elixir), invoked from
`BrokerLogAdapter.finalize/2`. Writes via a small HTTP call to a new
service-role endpoint on platform-api (see PR plan track A).

### 2. Reflector

Given a finished run's transcript + outcome, produce 1–N candidate
memory items by prompting an LLM with a fixed-format instruction:
"what did we learn about this workspace, this repo, this kind of task,
this failure mode? Output one JSON object per memory." The Reflector
runs *after* the run completes; it doesn't block the user-visible
turn loop.

Two execution shapes considered:

| Shape | Pros | Cons |
|---|---|---|
| Inline in `finalize/2` (same orchestrator process) | Simple. No new infra. | Adds a model call to every run's latency before next pickup. |
| Out-of-band via a `scheduled_task` row enqueued on finalize | Independent scaling. Reflector failures don't poison the run record. | One more hop. |

**Decision:** out-of-band. Enqueue a `scheduled_task` row tagged
`{kind: "reflection", source_run_id: <run>}` and let the existing
scheduler pick it up on the next tick. This reuses fully-built infra,
makes reflection retryable, and keeps the agent-run hot path clean.

### 3. Memory Retriever (tool-call primary, minimal pinned injection)

Two complementary surfaces, not one.

**Primary: `memory.search` tool the agent calls on demand.**

The agent decides when historical context matters and what to ask for.
A tool call lets the agent issue a focused query ("did we have a
problem with this specific file before?") rather than receiving the
same K memories every turn whether they're relevant or not.

Tool spec (per the existing platform tool model — registered alongside
`work_items.read`, `plans.read`, etc.):

```
memory.search({
  query: string,                        // natural-language query
  scope?: "workspace" | "agent" | "task",  // narrowing
  importance_min?: number,              // 1–10, default 1
  limit?: number                        // default 5, max 20
}) -> {
  results: Array<{
    id: string,
    content: string,
    importance: number,
    scope: string,
    tags: Record<string, unknown>,
    source_run_id: string | null,
    event_time: string
  }>
}
```

Under the hood: calls `searchMemoryItemsHybrid` (Track A2) with the
agent's `workspace_id`, optional `agent_id` narrowing, the workspace's
configured embedding provider for the query embedding, and the existing
RLS path so the agent only ever sees memories its workspace owns.

**Secondary: pinned facts in the system prompt.**

A small bounded section (≤ 3 items, ≤ 300 tokens total) prepended to
the system prompt at session start, populated from the workspace's
highest-`importance` `long_term`-scope memories. These are the "never
forget" facts ("this repo uses pnpm, not npm"; "tests need
`DATABASE_URL`"). Their job is to avoid a cold start where the agent
doesn't know it has a memory store to call.

The pinned section is the only mandatory injection. Everything else is
opt-in via the tool.

**Why this shape over every-message injection:**

| Tradeoff | Every-message injection | Tool-call + pinned (chosen) |
|---|---|---|
| Per-turn token cost | High and constant | Low baseline; spikes only when the agent searches |
| Query specificity | Same query / same K every turn | Specific to whatever the agent is working on now |
| Agent agency | None | Full — search multiple times with different queries |
| Cold start | Memories always visible | Pinned facts seed; tool description tells the agent it exists |
| Pollution risk in large memory stores | High (always picks N "somewhat related" items) | Low (only searches when relevant) |

The cost: agents that don't know to search will run blind. We mitigate
two ways — pinned facts seed the most-load-bearing context, and the
agent's system prompt explicitly says "you have access to a workspace
memory store via `memory.search` — use it when you need historical
context, prior decisions, or known gotchas."

**Cross-agent visibility (settled).** Memories with `agent_id IS NULL`
are workspace-wide and returned to any agent in the workspace.
Memories with a non-null `agent_id` are returned only when the searcher's
`agent_id` matches OR the search is explicitly scoped `workspace`.
The Reflector (Component 2) decides per-memory whether the insight is
agent-specific or workspace-wide — failure modes ("CI is flaky on
this branch") usually workspace-wide; agent-personality observations
("the planning agent prefers single-PR-per-task framing") agent-specific.

Lives in: **parallel-agent-platform** (`apps/api`) for the tool handler
and the pinned-prompt block; the runtime sees `memory.search` as just
another tool spec via the existing tool-registry path, no special
runtime-side support required.

### 4. Skill Distiller

A nightly (or per-N-runs) `scheduled_task` that:

1. Reads recent `memory_items` for a workspace where
   `scope='run_summary'` and `importance >= threshold`.
2. Clusters them (initially: just FTS-similarity grouping; pgvector
   k-means later).
3. For each cluster, prompts an LLM: "is there a reusable skill in
   these notes? If yes, output the skill body."
4. Opens a PR against the workspace's bound repo (via the existing
   GitHub integration) adding `.codex/skills/<slug>.md`.

The PR is the governance step. Humans review before the skill becomes
visible to future agent runs. This matches the Symphony-trusted-environment
principle and prevents agents from silently re-writing their own
instructions.

Depends on: `agent-skills-platform-scope.md` landing first. Until then,
the Distiller can be stubbed (writes the candidate to a
`run_summary`-scoped memory tagged `candidate_skill=true` and stops).

### 5. Governance + observability

- **Audit log.** Every memory write records `source_run_id`. Every
  skill-candidate PR links the runs that produced it.
- **No automatic deletion.** Use `is_deleted` soft-delete plus
  `supersedes_id` lineage. Memory eviction is a separate, deliberate
  process — out of initial scope.
- **Per-workspace memory budget.** Future: cap at N memories per
  workspace, evict lowest-importance. Not in v1.
- **Privacy.** Memories are workspace-scoped, RLS-enforced today.
  No cross-workspace federation.

## Boundary calls worth knowing

**Workspace-scoped, not user-scoped.** A workspace member's onboarding
intuition, repo conventions, and CI patterns belong to the workspace.
A user's personal preferences (preferred phrasing, response length)
are out of scope — that's Hermes's home turf, and we don't have a
single-user surface to make it useful here.

**Skills via PR, not auto-write.** The Hermes blueprint includes
autonomous skill creation. Per the Symphony "preview / trusted
environment" stance and the project's general "don't let agents poison
their own instructions" intuition, skill candidates surface as PRs.
The agent proposes; humans merge.

**No new tables.** Everything we need exists. `memory_items` is the
storage. `scheduled_task` is the cron. The (in-flight) skills table is
the destination for distilled skills. New work is wiring, not schema.

**One memory provider per workspace.** Embeddings cost money. The
workspace's primary model-provider credential is the default embedding
provider (e.g., OpenAI workspace → `text-embedding-3-small`). Workspaces
with only local-runtime providers run without embeddings and fall back
to FTS-only retrieval — `memory_hybrid_search` already handles the
null-embedding case.

**Reflection is bounded.** Reflector produces ≤ K memories per run (K=5
proposed). A run that's 500 turns long still yields ≤ 5 memory rows.
Keeps storage growth roughly linear in runs, not in turns.

## Decisions and open questions

### Decided

**Cross-agent memory sharing.** Yes — workspace-wide by default.
Memories with `agent_id IS NULL` are visible to any agent in the
workspace; memories with a non-null `agent_id` are filtered to the
matching agent (or surfaced via an explicit `scope=workspace` search).
The Reflector (Component 2) decides per-memory which it is — most
workspace knowledge (CI quirks, repo conventions, deployment
processes) is workspace-wide; agent-personality observations stay
agent-specific.

**Memory delivery is tool-call primary, not always-on prompt
injection.** See Component 3 above. Agents call `memory.search` when
they need historical context. A small pinned block in the system
prompt seeds the most-load-bearing facts so the agent isn't blind
on cold start.

**Provider rotation triggers a user-visible warning.** Embedding
vectors are provider-specific (OpenAI 1536-dim ≠ Anthropic future
embedding ≠ Voyage). When a user changes an agent's provider in the
Settings UI, we surface a modal:

> Changing this agent's provider will stop new embeddings from being
> generated by the previous provider. The agent's existing memories
> stay searchable via full-text search (no semantic similarity), and
> new memories saved after this change will use the new provider.
> Memories created with two providers cannot be ranked against each
> other on similarity — only on text match.
>
> [ Cancel ]   [ Change provider ]

`memory_hybrid_search` already degrades cleanly when embeddings
disagree (FTS-only ranking), so this is a UX warning, not a
correctness fix. We do not attempt to re-embed historical memories on
provider change — too expensive, may not be possible if the old
provider's credential is revoked.

> Future work, not in scope here: a provider-agnostic embedding layer
> (likely a small open-source model run via local-runtime-helper, or
> a Voyage/Cohere shared default) so workspace-wide search remains
> coherent across provider changes. Worth a separate scope doc once
> the cost trade-off is clearer.

### Still open

1. **Workspace-level "skill packs."** If a user works across N repos
   in one workspace, do skills live per-workspace or per-repo? Defer
   to the skills scope; not this doc's call.
2. **Cost cap.** Embedding + reflection model calls have a real dollar
   cost. Add per-workspace caps before turning this on widely (Track E1).
3. **Pinned-memory selection.** Top-N by importance is the v1 rule, but
   importance is set by the LLM during reflection — quality varies. May
   need an explicit "pin this" user action in the memory inspector UI
   (Track C3) once we see how scores distribute in real workspaces.

## Out of scope

- Personal-user modeling (Hermes's home territory)
- Cross-workspace memory federation
- Auto-merged skill changes
- Memory eviction / forgetting (separate later doc)
- TUI / messaging-gateway surfaces (parallel-agent has a web UI)
- Replacing the agent runner with a Hermes runner (the Symphony+Hermes
  comparison's Option C — explicitly not the path here)
